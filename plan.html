smallworld

The algorithmic photography angle finder. Computes optimal camera positions from terrain geometry and composition principles.

No tool today algorithmically analyzes terrain to find optimal photography viewpoints using composition principles. Every technical component exists independently. The integration is the innovation.

What It Does

A photographer or drone pilot specifies a location and search radius. smallworld analyzes the terrain geometry and computes the objectively best camera angles — positions and orientations that produce compositions following the golden ratio, rule of thirds, fibonacci spirals, leading lines, and other principles of visual beauty.

It also accepts a style reference (a painting, photo, or artwork) and finds terrain that geometrically resembles the reference when viewed from the right angle.

Results are rendered as photorealistic previews with GPS coordinates, altitude, and heading — ready to navigate to and shoot.

The Key Insight

Traditional approach: generate millions of random camera angles, render each one, score for beauty. Computationally impossible.

Our approach: composition rules are geometric constraints. If you know where terrain features are in 3D and where you want them in the 2D frame, the camera position is a solvable equation — not a search problem. This is the inverse of the Perspective-n-Point problem from computer vision.

10,000,000,000 possible viewpoints (brute force)
↓ Deterministic computation from geometry
~500 candidates, all compositionally valid
↓ Score for non-composition beauty
~50 beautiful, well-composed viewpoints
↓ Render + neural aesthetic scoring
~20 top viewpoints
↓ CMA-ES fine-tuning
~10 optimized final viewpoints delivered

Phase-by-Phase Algorithm

Phase 1: Terrain Feature Extraction

When a user clicks the map and sets a radius, we download elevation data from AWS Terrain Tiles (free, unlimited, global). Each tile is an RGB PNG encoding elevation. We decode it to a NumPy grid at ~30–75m resolution.

Terrain Derivatives

DerivativeWhat it findsMethod
SlopeSteepnessFirst partial derivatives (Horn’s formula)
Profile curvatureCliff edges, convex/concave breaksSecond derivative along slope
Plan curvatureRidge crests, valley bottomsSecond derivative ⊥ to slope
LaplacianDramatic terrain changesSum of second partial derivatives

Feature Extraction

  • Peaks: Local maxima filtered by topographic prominence (minimum descent to reach higher terrain). Separates landmarks from noise.
  • Ridgelines: Invert the DEM, run D8 flow accumulation. Water paths in the inverted map = ridgelines in reality. An elegant hydrology trick.
  • Cliffs: Cells where |profile curvature| exceeds the 95th percentile.
  • Streams: D8 flow accumulation on normal DEM. High upstream drainage = stream channel.
  • Lakes: Fill depressions; cells where filled DEM exceeds original by a threshold.
  • Saddle points: Strongly negative Gaussian curvature — terrain curves up in one direction, down in another.

Interest Map

All features combined into a weighted heatmap. User adjusts weights via sliders:

interest = (
    w_peaks  × distance_decay(peaks)  +
    w_ridges × ridgelines             +
    w_cliffs × normalize(|profile_curvature|) +
    w_water  × distance_decay(streams | lakes) +
    w_relief × normalize(local_elevation_range)
)

Time estimate: ~5 seconds for 20km radius.

Phase 2: Scene Grouping

Cluster features into photographable “scenes” — groups that could appear together in one frame:

  • Peak + nearby lake → mountain reflection shot
  • Two peaks + connecting ridge → dramatic skyline
  • Cliff face + valley floor → vertical drama
  • Stream confluence + surrounding peaks → compositional anchor
  • Ridge + distant peak behind → depth/layering

Phase 3: Deterministic Camera Computation

This is the core innovation. Composition rules define where features should appear on screen. Feature positions are known in 3D. The camera position that places features at those spots is a solvable equation, not a search.

Composition Templates

Rule of Thirds — primary feature at one of four power points:

┌───────&boxdh;───────&boxdh;───────┐ │ │ │ * │ ← Peak at (2W/3, H/3) ├───────&boxcr;───────&boxcr;───────┤ │ │ │ │ ├───────&boxcr;───────&boxcr;───────┤ │ * │ │ │ ← Lake at (W/3, 2H/3) └───────&boxdh;───────&boxdh;───────┘

Golden Ratio — features at φ intersections, horizon at 0.618 × H.

Leading Line — camera behind a ridge, looking along it toward the subject:

┌────────────────────┐ │ * peak │ │ ⁄ │ │ ⁄ ridge │ │ ⁄ │ │ ⁄ │ └───⁄────────────────┘

Symmetry — camera on perpendicular bisector of two features, centered horizon.

Fractal Dimension Distance

The skyline is dominated by ridgelines — already extracted from the DEM. We compute each ridgeline’s fractal dimension directly from its elevation profile at multiple smoothing scales. Viewing distance determines which scale you perceive, so we can solve for the distance that produces D ≈ 1.3:

for ridge in ridgelines:
    profile = elevation_along(ridge)
    for scale in [50, 100, 200, 500, 1000]:  # meters
        fd_at_scale[scale] = box_counting_fd(smooth(profile, scale))

    # find scale where D ≈ 1.3
    optimal_scale = interpolate(fd_at_scale, target=1.3)
    optimal_distance = optimal_scale * image_width_px / fov_radians

    # camera perpendicular to ridge at that distance
    camera = ridge.midpoint + ridge.normal * optimal_distance

No rendering needed — the ridgeline geometry tells us exactly where to stand for the optimal skyline complexity.

Solving for Camera Pose

Pitch is computed directly from desired horizon placement:

pitch = arctan((desired_horizon_y - H/2) / focal_length)

Position + yaw from feature placement (inverse PnP): with 2 features assigned to 2 screen positions and pitch known, we have 4 equations and 4 unknowns. A determined system with 1–2 solutions.

Each scene × each composition template yields ~20–30 camera poses. All compositionally valid by construction. Total: ~300–500 candidates in under 1 second.

Phase 4: Fast Beauty Scoring

Composition is already guaranteed. Score only for non-compositional beauty using terrain math alone (no rendering):

Viewshed Richness
How much interesting terrain is visible. Visibility mask × interest map.
Viewpoint Entropy
Shannon entropy of elevation bins in visible area. Diverse terrain = interesting view.
Skyline Fractal Dim
Cast rays across FOV, record max elevation angle per ray → 1D skyline profile. Box-counting gives fractal dimension. Score peaks at D ≈ 1.3. Free — reuses viewshed ray-casting.
Prospect-Refuge
Balance of open vista (prospect) and nearby enclosure (refuge). Harmonic mean of viewshed area vs. terrain above horizon.
Depth Layering
Foreground/midground/background balance. Score evenness of terrain at three distance zones.
Mystery Score
Ratio of “almost visible” interesting terrain to total visible terrain. Valleys curving out of sight score higher.
Water Visibility
Amount of visible water features. Strongly correlated with scenic beauty ratings (Schirpke, R²=0.72).

Time estimate: ~15 seconds for 500 candidates.

Phase 5: Render + Neural Scoring

Top ~50 candidates rendered via CesiumJS headless (Puppeteer screenshots). Each image scored by:

  • LAION Aesthetics v2.5 — neural beauty score (1–10), pip-installable, ~50ms/image.
  • CACNet composition verification — confirms the render matches the intended composition type.

Note: skyline fractal dimension is already computed in Phase 4 from DEM data alone — no rendering needed.

Time estimate: ~30 seconds.

Phase 6: CMA-ES Refinement

Top 10 candidates fine-tuned with CMA-ES (Covariance Matrix Adaptation Evolution Strategy). Empirically proven to achieve 3× better aesthetic improvement than gradient descent for camera pose optimization.

It maintains a population of 8 candidate viewpoints, evaluates each, and adapts its search distribution to home in on the optimal angle. ~50 evaluations per candidate.

Time estimate: ~2 minutes (500 total renders).

Phase 7: Optimal Lighting Time

For each final viewpoint, compute the best time of day for dramatic light. The sun is a directional light source, and shadow casting on a DEM is the same ray-casting we already use for viewshed — just from the sun’s direction.

Sample sun positions every 15 minutes from sunrise to sunset (~50 samples via astral or pvlib). For each, cast shadows on the DEM and score the shadow pattern within the camera’s visible area.

Shadow/Light Ratio
~35% shadow is ideal. Too little = flat noon light. Too much = underexposed.
Sun Elevation
Low sun (~8°) creates long, dramatic shadows. Golden hour exists for a reason.
Rake Angle
Light perpendicular to cliff faces reveals every ridge and crevice. Head-on light flattens everything.
Peak-Lit / Valley-Shadow
The classic landscape look — illuminated summits with shadowed valleys. Score by elevation×light correlation.

Output per viewpoint: Best time (“7:14 AM”), lighting description (“eastern light rakes cliff face, peaks lit with valley in shadow”), secondary time, and a mini timeline of scores throughout the day. The best achievable shadow score also feeds back into overall viewpoint ranking.

Time estimate: ~2–3 seconds per viewpoint (reuses viewshed infrastructure).

Phase 8: Enhancement + Export

CesiumJS renders passed through Nano Banana 2 (Gemini 3.1 Flash, free API) for photorealistic previews. Style reference applied via IP-Adapter + ControlNet.

Drone flight path export: Top viewpoints exported as a Litchi-compatible CSV. Columns map directly from our computed camera poses: lat/lng → position, altitude (MSL), heading → yaw, gimbal pitch → camera tilt. Viewpoints ordered by nearest-neighbor TSP to minimize flight distance. User imports CSV into Litchi, flies the mission — the drone navigates to each viewpoint, sets up the exact shot, and auto-triggers the shutter.

Style Reference Pipeline

The style reference must change what terrain we look for, not just re-rank final candidates. The solution: contour topology matching.

Step 1: Structural Fingerprint

Extract the geometric skeleton of the reference image via HED edge detection (~50ms). Compute a ~100-dimensional fingerprint: curvature histogram per quadrant, orientation histogram per quadrant, spatial density, parallelism, dominant direction.

Step 2: Terrain Patch Matching

Slide a window across the DEM. For each patch, compute contour line properties (spacing, curvature, parallelism, orientation, sinuosity). Compare against reference fingerprint. Matching patches become style-driven POIs.

Step 3: Deterministic Camera from Surface Normals

For each matching patch, the terrain surface normal tells us where to stand. Distance determined by feature scale vs. reference scale. Height from edge vertical distribution in reference. This is computation, not search.

Step 4: Gradient Refinement

Project contour lines onto camera plane (matrix multiplication), compute fingerprint, compare. Iterate via numerical gradient ascent. 50 iterations × 10 perturbations = 500 evaluations at microseconds each = under 1 second.

Step 5: Rendered Comparison

Top 20 style-matched candidates rendered and compared via CLIP embeddings + LPIPS. Now meaningful because candidates were selected for geometric similarity.

Agent / Conversational Interface

The agent wraps the pipeline — it doesn’t sit inside it. The core math runs at native speed with zero LLM overhead. The agent orchestrates phases via tool calls and adds intelligence at the boundaries where natural language understanding matters.

Where the agent doesn’t go: Terrain analysis, inverse PnP, beauty scoring, CMA-ES — all pure NumPy/SciPy math. An LLM there would add latency for zero benefit. The agent adds value at five specific touchpoints where structured computation can’t handle ambiguity.

1. Natural Language → Filter Config

User says: “Find dramatic cliff shots near Yosemite, something with depth and a strong foreground.”

The agent translates this to pipeline parameters — location, feature weights, beauty weights, composition preferences, mode. Replaces manual slider adjustment with a single sentence.

2. Result Refinement Loop

After results return, the user says: “These are too far from the cliff edge. I want to feel like I’m right on the edge looking down.”

The agent interprets this as parameter adjustments (min slope at camera, max distance from cliff, downward pitch preference) and re-runs the pipeline. Each refinement cycle takes 30–60 seconds. Conversation context is maintained so each iteration builds on the last.

3. Result Descriptions

For each viewpoint, the agent generates a photographer-friendly narrative:

“You’re standing on a granite ledge at 2,847m, 40m back from the cliff edge. Half Dome fills the upper-right third of the frame with El Capitan’s face in the left midground. A natural leading line runs from the pine tree cluster at your feet toward the valley floor. Best light at 6:42 AM when low eastern sun rakes across the cliff face.”

Generated in parallel for all results — doesn’t block the pipeline.

4. Style Reference Interpretation

When a user uploads a reference image, the agent adds semantic understanding on top of the geometric fingerprinting — not just “find terrain with similar curves” but “find terrain that would produce similar dramatic impact.” It augments the contour topology matcher with high-level compositional intent.

5. Multi-Location Itinerary Planning

For multi-day photography trips, the agent chains pipeline runs across multiple coordinates, cross-references optimal lighting times, and plans logistics:

Day 1 — Lauterbrunnen Valley
  5:45 AM  Männlichen ridge (golden hour, Eiger north face)
  7:30 AM  Kleine Scheidegg (morning side-light on Jungfrau)
  5:15 PM  Stechelberg cliff edge (sunset rake on valley walls)

Day 2 — Zermatt
  6:00 AM  Gornergrat viewpoint (Matterhorn dawn)
  ...

Route: 127km driving, ordered by time-of-day constraints.
Litchi CSV for drone shots attached.

The Mathematical Foundation

Fractal Dimension ≈ 1.3

Humans consistently prefer landscapes with fractal dimension around 1.3–1.5 (Sprott 2003, Hagerhall 2004). This matches clouds, coastlines, tree canopies — patterns our visual system evolved to process fluently.

No rendering needed. The skyline is computable directly from the DEM: cast rays across the FOV, record the max elevation angle per ray — that 1D profile IS the skyline. Box-counting gives the fractal dimension. This reuses viewshed ray-casting, so it’s essentially free.

Also deterministically computable. Ridgeline fractal dimension at multiple smoothing scales lets us solve for the viewing distance where D ≈ 1.3 — another geometric constraint for camera placement, not a search-and-filter metric.

Prospect-Refuge Theory

Appleton (1975): we prefer views balancing open vistas (prospect — spot predators) and nearby enclosure (refuge — hide). Cliff edges overlooking valleys with mountains behind = perfect balance.

Kaplan & Kaplan Preference Matrix

FactorDefinitionComputation
CoherenceVisual order, organizationTexture homogeneity, moderate image entropy
ComplexityRichness, number of elementsEdge density × color entropy (optimal is intermediate)
LegibilitySense of place, navigabilityVisible landmarks + open traversable areas
MysteryPromise of more informationNear-viewshed-boundary interesting terrain / total visible

Viewpoint Entropy

Vázquez et al. (2001): Shannon entropy of the visibility distribution. H = −Σ pi × log(pi). Diverse visible terrain = high entropy = interesting view.

Scenic Beauty Estimation

Schirpke et al.’s GIS model (R²=0.72): shape complexity (+), landscape diversity (+), water/lakes/glaciers (++), high-altitude vistas (+), infrastructure proximity (−).
